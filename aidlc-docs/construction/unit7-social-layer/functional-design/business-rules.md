# Business Rules — Unit 7: Social Layer

## BR-U7-01: Формат username
- Username должен соответствовать regex `^[a-z0-9_]{3,20}$`
- Только строчные латинские буквы, цифры и underscore
- Длина: 3–20 символов
- Валидируется клиентски перед signUp
- Принудительно в БД через CHECK constraint
- **Кириллица запрещена**

## BR-U7-02: Уникальность username
- Username должен быть уникальным в системе
- Обеспечивается UNIQUE constraint на `profiles.username`
- При дубликате триггер вернёт ошибку → frontend показывает "Этот логин уже занят"

## BR-U7-03: Автосоздание профиля
- При любой регистрации (signUp) автоматически создаётся профиль через PostgreSQL триггер `on_auth_user_created`
- Username берётся из `auth.users.raw_user_meta_data->>'username'`
- Если триггер не срабатывает — профиль не создаётся (fallback: email в Navbar)

## BR-U7-04: Приватность библиотеки
- `is_library_public = true` (default) → библиотека и статистика видны всем авторизованным пользователям
- `is_library_public = false` → возвращаются пустые mediaItems и пустая статистика
- **Тоггл не реализован в Unit 7** — заложен для будущего юнита настроек профиля

## BR-U7-05: Доступ к чужим данным
- Профили читаются через обычный Supabase клиент (RLS разрешает `SELECT TO authenticated`)
- `media_items`, `seasons`, `episodes` чужих пользователей читаются **только через service role** на сервере
- Клиентский код не получает доступа к чужим данным напрямую

## BR-U7-06: Поиск пользователей
- Поиск выполняется по `ilike` (case-insensitive) по полю `username`
- Результаты ограничены 20 записями
- Пустой запрос → возвращается пустой массив (без запроса к БД)

## BR-U7-07: Профиль текущего пользователя в Navbar
- В Navbar отображается `@username` текущего пользователя
- Ссылка ведёт на `/profile/[username]`
- Если профиль не найден (старый пользователь без профиля) — отображается email как fallback
